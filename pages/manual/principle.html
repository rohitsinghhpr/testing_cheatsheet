Here are the 7 principles of manual testing explained simply, along with examples to illustrate each one:

1. Testing Shows Presence of Defects

  • Principle: Testing can prove that bugs exist in a system, but it cannot prove that there are no bugs.
  • Example: If a tester finds multiple bugs in a new feature, it indicates defects are present. However, even if all test cases pass, it's impossible to guarantee there are no hidden issues.

2. Exhaustive Testing is Impossible

  • Principle: Testing every possible scenario is impractical due to the vast number of inputs, outputs, and paths. Instead, testers focus on the most critical and common scenarios.
  • Example: For a login screen, instead of testing all possible combinations of characters in the password, testers might only check common cases like empty input, valid and invalid passwords, and edge cases (like max character length).

3. Early Testing

  • Principle: Testing should begin as early as possible in the software development lifecycle to find and fix defects sooner, which reduces costs.
  • Example: Reviewing requirements before development starts can help spot unclear requirements that could lead to defects later, saving time on fixing bugs after development.

4. Defect Clustering

  • Principle: Most defects are typically found in a small number of modules or areas. This is often due to code complexity, new features, or other factors.
  • Example: In an e-commerce application, if bugs are frequently found in the checkout process, testers will pay more attention to that area during future testing.

5. Pesticide Paradox

  • Principle: Repeating the same tests will eventually stop finding new defects, much like pests becoming immune to the same pesticide. Tests need to be reviewed and updated regularly to remain effective.
  • Example: If a tester uses the same test cases for each software version, eventually those test cases will fail to uncover new defects. Creating new test cases and modifying existing ones helps find fresh issues.

6. Testing is Context Dependent

  • Principle: Testing varies based on the type of application. Different applications require different types of testing approaches, strategies, and techniques.
  • Example: A banking app requires high-security and reliability testing, while a social media app might focus more on usability and performance to handle many users.

7. Absence of Errors Fallacy

  • Principle: Even if no defects are found, it doesn’t mean the software is ready if it doesn’t meet user needs or if it was built based on incorrect requirements.
  • Example: Suppose a payroll system was developed and tested without any issues. If it fails to calculate overtime correctly according to company policy, it doesn’t matter that there were no bugs; it’s still unusable for users.

These principles guide testers in prioritizing efforts, ensuring they focus on creating meaningful, efficient, and effective testing processes.
