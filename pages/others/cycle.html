SDLC (Software Development Life Cycle)
The Software Development Life Cycle (SDLC) is a structured process that guides the development of software. It consists of distinct phases, ensuring that software is developed in a systematic, efficient, and predictable manner. The typical phases of SDLC are:
  • Requirement Gathering and Analysis: Understanding and documenting the requirements from stakeholders.
  • Design: Creating software architecture, system design, and UI/UX design.
  • Implementation (Coding): Writing the actual code based on the design.
  • Testing: Verifying the softwareâ€™s functionality, performance, and security.
  • Deployment: Releasing the software to users or production environments.
  • Maintenance: Addressing bugs, updating the software, and improving it post-release.
SDET (Software Development Engineer in Test)
An SDET is a role focused on both software development and quality assurance. SDETs are responsible for creating test frameworks, automating test cases, and ensuring that the product meets quality standards through both manual and automated testing. Their role integrates the following:
  • Test Automation Development: Building test automation scripts and frameworks to increase test efficiency and coverage.
  • Collaboration with Developers: Working closely with developers to ensure testability of code and integrating testing into the development process.
  • Performance and Security Testing: Ensuring that software performs well under load and is secure from vulnerabilities.
  • Continuous Integration: Implementing automated tests into the CI/CD pipeline for continuous validation of code changes.
Bug Life Cycle
The Bug Life Cycle, also known as the Defect Life Cycle, defines the various stages a bug goes through from identification to resolution. Here are the common stages:
  • New: The bug is reported but not yet confirmed.
  • Assigned: A developer is assigned to investigate and fix the bug.
  • Open: The developer starts working on the bug fix.
  • Fixed: The developer has fixed the bug and it is ready for testing.
  • Retesting: The QA team tests the fix to ensure the bug is resolved.
  • Verified: The QA confirms that the bug is fixed and no longer present.
  • Closed: The bug is marked as resolved and the issue is considered closed.
  • Reopened: If the bug persists after being marked as fixed, it may be reopened for further investigation.

Defect status [NAFCRR]

  • new - when a new defect is logged and posted for the first time.
  • assigned - once the bug is posted by tester, the lead (TL) of tester approves the bug and assign the bug to the developer team.
  • open - the developer starts analyzing and works on the defect.
  • pending retest - fixed defect dev gives to qa.
  • retest - check weather the defect is fixed or not.
  • verified - the tester re-tests the bug after it got fixed by the dev. if there is no bug detected in soft, then the bug is fixed and status assigned is verified.
  • reopen - if the the bug persists even after the developer has fixed the bug, the tester changes the status to "reopened". then once again the bug goes through life cycle.
  • closed - if bug is no longer exits then tester assign the status "closed".
  • duplicate - if the defect is repeated twice.
  • rejected - if dev feels the defect is not a geniue defect.
  • deferred - if the present bug is not a prime priority and if it is expected to get fixed in the next release.
  • not a bug - if bug does not affect the functionality of the app then status assigned to a bug is "not a bug"
