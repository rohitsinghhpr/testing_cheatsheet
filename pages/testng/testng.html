Annotations of testng.
  1. @Test: Marks a method as a test method.
  2. @BeforeSuite : Runs before all tests in the suite.
  3. @AfterSuite : Runs after all tests in the suite.
  4. @BeforeTest : Runs before any test method belonging to the classes insidethe tag.
  5. @AfterTest : Runs after all test methods belonging to the classes inside the tag.
  6. @BeforeClass : Runs before the first method in the current class is invoked.
  7. @AfterClass : Runs after all methods in the current class have been invoked.
  8. @BeforeMethod : Runs before each test method.
  9. @AfterMethod : Runs after each test method.
  10. @BeforeGroups : Runs before the first test method in the specified group isinvoked.
  11. @AfterGroups : Runs after the last test method in the specified group isinvoked.
  12. @DataProvider : Marks a method as a data provider.
  13. @Factory : Marks a method as a factory that returns objects that will be usedas TestNG tests.

Types of Assertions

In TestNG, assertions are used to validate that the expected results match the actual results during testing. Assertions help identify issues and ensure the stability of the code. TestNG provides two types of assertions:

1. Hard Assertions (Assert Class)

Hard assertions halt the execution of the test immediately if the assertion fails. Common methods in the org.testng.Assert class include:

  • assertEquals(actual, expected)

    • Validates if two values are equal.
    • Example: Assert.assertEquals(actual, expected);

  • assertNotEquals(actual, expected)

    • Validates if two values are not equal.

  • assertTrue(condition)

    • Checks if the given condition is true.
    • Example: Assert.assertTrue(isElementVisible);

  • assertFalse(condition)

    • Checks if the given condition is false.

  • assertNull(object)

    • Validates if the given object is null.

  • assertNotNull(object)

    • Validates if the given object is not null.

  • assertSame(actual, expected)

    • Validates if two references point to the same object.

  • assertNotSame(actual, expected)

    • Validates if two references do not point to the same object.

2. Soft Assertions (SoftAssert Class)

Soft assertions allow the test to continue executing even if an assertion fails. Results are collected and reported at the end of the test. The org.testng.asserts.SoftAssert class is used for soft assertions.

Steps to use SoftAssert:

  1. Instantiate the SoftAssert class.
  2. Use assertion methods (assertEquals, assertTrue, etc.).
  3. Call the assertAll() method at the end to collate results and mark the test as passed or failed based on all assertions.

Example:

import org.testng.asserts.SoftAssert;

public class SoftAssertExample {
    @Test
    public void testSoftAssertions() {
        SoftAssert softAssert = new SoftAssert();
        softAssert.assertEquals(1, 2, "First assertion failed");
        softAssert.assertTrue(false, "Second assertion failed");
        // More assertions can go here.
        
        softAssert.assertAll(); // Must be called at the end
    }
}

import org.testng.annotations.DataProvider;
import org.testng.annotations.Test;

public class DataProviderExample {

/*
* Internal DataProvider Example
* The DataProvider is defined within the same class as the test method.
*/
@DataProvider(name = "internalDataProvider")
public Object[][] provideDataInternal() {
return new Object[][] {
{"Alice", 30},
{"Bob", 25},
{"Charlie", 35}
};
}

@Test(dataProvider = "internalDataProvider")
public void testInternalDataProvider(String name, int age) {

}

@Test(dataProvider = "externalDataProvider", dataProviderClass = ExternalDataProvider.class)

System.out.println("Testing with: " + username + " - Role: " + role);
// Test logic using provided data
}
}

Listeners

1. Listeners

  • Listeners are classes that implement one or more TestNG listener interfaces (e.g., ITestListener, IInvokedMethodListener, ISuiteListener).
  • They allow you to intercept and act on various test lifecycle events.
Common Listener Interfaces
  • ITestListener: For events related to individual test methods (e.g., onTestStart, onTestSuccess, onTestFailure).
  • ISuiteListener: For suite-level events (onStart and onFinish for the entire suite).
  • IInvokedMethodListener: For events before and after each test method invocation (beforeInvocation, afterInvocation).
  • IConfigurationListener: For events related to configuration methods (e.g., beforeConfiguration).

XMl Tags and Attribute

Summary of Main Tags and Attributes

  • <suite>: Defines the entire suite of tests.

    • Attributes: name, verbose, parallel, thread-count, time-out, etc.

  • <listeners>: Adds listeners for events like test start, finish, and failure.

    • Attributes: class-name (class name of the listener).

  • <test>: Defines a group of classes/methods as a single test unit.

    • Attributes: name, parallel, thread-count, time-out, preserve-order, etc.

  • <groups>: Specifies test groups to include or exclude.

    • Attributes for <include>/<exclude>: name (name of the group).

  • <packages>: Allows the inclusion of all classes within a package.

    • Attributes for <package>: name (name of the package).

  • <classes>: Lists the classes to include in the test.

    • Attributes for <class>: name (fully qualified class name).

  • <methods>: Controls specific methods to include or exclude within a class.

    • Attributes for <include>/<exclude>: name (name of the method).

  • <parameter>: Defines parameters that can be accessed by @Test methods.

    • Attributes: name, value.
